## 1.1.0 功能变更要求：

4. 应用所有读取DB的 SELECT 语句，包括使用AI动态生成的SQL语句，每一次在真正执行SQL select 之前，根据select脚本的内容，进行以下操作：
    1) 系统在真正执行 SQL 查询之前，必须先对 SQL 做静态解析，识别本次查询实际读取的底层数据库表和数据列，并对这些资源逐一调用 'QueryVisibility' 接口。
        a) 仅允许执行只读 SELECT 查询。任何 INSERT、UPDATE、DELETE、MERGE、DROP、ALTER、CREATE、TRUNCATE、EXEC 等非只读 SQL，直接拒绝执行。
        b) 系统必须基于 SQL 解析结果，得到“底层基表”和“底层基列”的唯一集合；只有在所有涉及资源都识别完成后，才允许进入 'QueryVisibility' 检查流程。若 SQL 无法被可靠解析，或无法准确识别所有底层基表/基列，则停止执行，并向 UI 显示错误：'Unable to determine all tables and columns referenced by the query.'
        c) 对于底层资源识别，按以下规则处理：
            I. 普通列引用：直接识别其所属基表和基列。
            II. 表别名、列别名：别名仅用于 SQL 书写，权限检查必须映射回真实基表名和基列名。
            III. SELECT * 或 table_alias.*：必须先根据当前数据库元数据展开为实际列清单，再按展开后的每一个真实列执行检查；如果无法展开，则拒绝执行。
            IV. JOIN：JOIN 两侧涉及的所有基表，以及 SELECT、JOIN ON、WHERE、GROUP BY、HAVING、ORDER BY 中引用的所有基列，都必须纳入检查。
            V. 表达式、函数、CASE、字符串拼接、类型转换、聚合函数：不对表达式结果本身做权限检查，但必须对表达式中引用到的所有底层基列执行检查。
            VI. 子查询、CTE、派生表：必须递归展开并追溯到最终的底层基表和基列，再执行检查。
            VII. 派生列、计算列、聚合结果列、别名结果列：如果其值来自一个或多个底层基列，则检查这些底层基列；如果不引用任何数据库列（例如纯常量），则不需要列级检查。
            VIII. 字面量、常量、SQL 参数：不需要执行 'QueryVisibility' 检查。
            IX. 重复引用的表或列只检查一次，按去重后的唯一集合调用接口。
        d) 系统要为每一个要读取的底层数据库表调用一次表级 'QueryVisibility'，为每一个要读取的底层数据列调用一次列级 'QueryVisibility'。
        e) 参数 'QualifiedName' 的组装方式如下：
            “QualifiedName 是 Azure SQL Database 资产在 Purview 中的唯一路径标识。其格式按层级拼接：
            mssql://{serverName}.database.windows.net/{databaseName}/{schemaName}/{tableName}/{columnName}
            如果查询表 QualifiedName 到 table 级：
            mssql://{serverName}.database.windows.net/{databaseName}/{schemaName}/{tableName}
            如果查询列，则在末尾追加列名：
            mssql://{serverName}.database.windows.net/{databaseName}/{schemaName}/{tableName}/{columnName}”
        f) 参数 'UserPrincipalName', 使用当前登录账户的 UserPrincipalName。
        g) 系统返回字符串'result_string'
    2) 为所有资源执行完 'QueryVisibility' 调用后，进行下述判断
        a) 如果所有返回结果只有 'ALLOW' 或 'INDETERMINATE' 或 'NOT_APPLICABLE', 则正常执行后续流程，不做任何变化或操作
        b) 如果返回结果中，有 'DENY' 则，停止后续流程，在用户 UI 界面中显示 'Current user is not allowed to read data:' + 'QualifiedName' 的值，如果有多个 'QualifiedName' 值，则使用 逗号分隔，将所有 'QualifiedName' 的值显示出来
        c) 如果返回结果中，没有'DENY'，但是 有 'MASK' ，则：
            I) 如果数据库表的返回是 'MASK'，则将搜索结果中，除去'ID','*-ID'或'*ID'列的其他所有列的所有 cell 中的内容，逐一调用 'Mask'接口，'QualifiedName'参数为该数据库表的 'QualifiedName' 值，'MaskInput'的值是 该 cell 的搜索结果。然后 使用 'Mask'接口 返回的结果字符串，替换 cell 中的元字符串。并不再检查数据列的'QueryVisibility' 返回，是否包括 'MASK'。
            II) 如果针对表格调用的'QueryVisibility'接口的返回是'ALLOW' 或 'INDETERMINATE' 或 'NOT_APPLICABLE', 但该表的某个或多个数据列的返回是 'MASK'，则将搜索结果中的该列的所有 cell 中的内容，逐一调用 'Mask'接口，'QualifiedName'参数为该 列的 'QualifiedName' 值，'MaskInput'的值是 该 cell 的搜索结果。然后 使用 'Mask'接口 返回的结果字符串，替换 cell 中的元字符串。
            III) 完成所有的 'MASK' 接口调用及 Cell 替换之后，继续后续流程。

7. 随后要求用户从界面上显示的客户列表中，选择一个客户，选择后点击“客户分析”按钮，点击客户分析按钮后，系统自动获取该客户过去的所有订单列表，展示在界面上。
8. 随后系统自动使用AI生成客户分析。系统将该客户的基本信息，客户过去的订单信息，以及 本页显示的product信息 输入AI。请AI帮忙对产品销售的可能性分析，限制在200字。将分析结果显示在界面上。
9. 随后要求用户在界面上从产品列表中选择产品（单选），选择完成后点击“产品趋势搜索”按钮，，系统首先组织搜索关键字，使用实时搜索互联网6周以内的与选定产品销售趋势相关的 top10 的US的英文新闻。首先将这些新闻，标题及链接显示在界面上。
10. 随后自动提取这些新闻内容的主要内容，将新闻内容，输入给AI，要求AI总结该产品短期过去及可预测的未来的销售趋势（200字）。完成后将销售趋势信息显示在界面上。
11. 之后系统等待用户点击“广告文案生成”按钮，点击后，系统使用AI生成广告文案（200字），向AI输入：1. 产品趋势的结合文字总结，2. 用户之前输入的客户描述文字，3. 从数据库获取的客户信息，由AI生成针对描述的客户客群的广告文案。显示在界面上。
12. 完成后，用户重置按钮。点击后，UI回到整个页面的登陆后的状态。即默认显示产品列表。
13. 所有的AI使用记录（包括前面所有环节）都会被记录用于后续的查询(由其他应用完成，不在本项目范围内)。
14. 系统无管理员后台维护功能。

深色，蓝色主色调，非报警内容不要用黄色或红色，报警优先用黄色。

页面主标题；
关键数据或核心指标；
主要按钮（CTA）；
当前状态、重要提示、关键结果；
模块标题和重要说明文字；

通过文字颜色、字重、背景层次、标签样式、边框或局部高亮，拉开信息层级，让用户进入页面后能快速看到最重要的内容。

在合适的位置加入 Tooltip、帮助说明、占位符、空状态文案、无数据提示，帮助用户快速理解页面内容和操作含义。
4. 排版与布局
优化信息层级、模块分组、留白、间距、对齐和视觉节奏；
调整标题、正文、说明、表单、按钮、列表、表格之间的层次关系，使重点更突出；
避免信息堆叠、页面拥挤、对齐混乱或主次不分；
优化按钮位置和操作分组，让常用操作更易发现，相关操作更贴近对应内容；
优化列表、表单、卡片、弹窗、筛选器、页头和操作栏的排版，提高可读性与可扫描性。
